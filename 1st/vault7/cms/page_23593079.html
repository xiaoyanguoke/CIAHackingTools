Tor

Tor is an encrypted anonymising network that makes it harder to intercept internet communications, or see where communications are coming from or going to.

In order to use the WikiLeaks public submission system as detailed above you can download the Tor Browser Bundle, which is a Firefox-like browser available for Windows, Mac OS X and GNU/Linux and pre-configured to connect using the anonymising system Tor.

Tails

If you are at high risk and you have the capacity to do so, you can also access the submission system through a secure operating system called Tails. Tails is an operating system launched from a USB stick or a DVD that aim to leaves no traces when the computer is shut down after use and automatically routes your internet traffic through Tor. Tails will require you to have either a USB stick or a DVD at least 4GB big and a laptop or desktop computer.

Tips

Our submission system works hard to preserve your anonymity, but we recommend you also take some of your own precautions. Please review these basic guidelines.

1. Contact us if you have specific problems

If you have a very large submission, or a submission with a complex format, or are a high-risk source, please contact us. In our experience it is always possible to find a custom solution for even the most seemingly difficult situations.

2. What computer to use

If the computer you are uploading from could subsequently be audited in an investigation, consider using a computer that is not easily tied to you. Technical users can also use Tails to help ensure you do not leave any records of your submission on the computer.

3. Do not talk about your submission to others

If you have any issues talk to WikiLeaks. We are the global experts in source protection – it is a complex field. Even those who mean well often do not have the experience or expertise to advise properly. This includes other media organisations.

After

1. Do not talk about your submission to others

If you have any issues talk to WikiLeaks. We are the global experts in source protection – it is a complex field. Even those who mean well often do not have the experience or expertise to advise properly. This includes other media organisations.

2. Act normal

If you are a high-risk source, avoid saying anything or doing anything after submitting which might promote suspicion. In particular, you should try to stick to your normal routine and behaviour.

3. Remove traces of your submission

If you are a high-risk source and the computer you prepared your submission on, or uploaded it from, could subsequently be audited in an investigation, we recommend that you format and dispose of the computer hard drive and any other storage media you used.

In particular, hard drives retain data after formatting which may be visible to a digital forensics team and flash media (USB sticks, memory cards and SSD drives) retain data even after a secure erasure. If you used flash media to store sensitive data, it is important to destroy the media.

If you do this and are a high-risk source you should make sure there are no traces of the clean-up, since such traces themselves may draw suspicion.

4. If you face legal action

If a legal action is brought against you as a result of your submission, there are organisations that may help you. The Courage Foundation is an international organisation dedicated to the protection of journalistic sources. You can find more details at https://www.couragefound.org.

iOS Triage Process

This describes the process needing to been done when a new iOS version is released.

Pre Setup

  1. Make a directory store output folders
    1. mkdir OUT
    2. set $OUT
      1. export OUT=`pwd`/OUT (PWD = mac command for present working directory)
  2. Grab the entire ios_manifest
    1. mkdir ios_repo and change into it
    2. repo init -u ssh://git@stash.devlan.net:7999/ios/ios_manifest
    3. repo sync
    4. verify your iOS manifest is on the right branch for the current version of iOS to be tested
      1. change into the .repo/manifests/ dir and verify you are on the correct manifest branch if not check out the right branch and re-sync
    5. set $IOS_REPO

      1. 'export IOS_REPO=<PATH>/ios_repo'

  3. Grab a testdylib for testing external to the IOS_REPO
    1. get testdylib_manifest
      1. repo init -u ssh://git@stash.devlan.net:7999/ios/testdylib_manifest
      2. repo sync
    2. cd into testdylib_repo/testdylib
    3. make
    4. CD ..
    5. set $TEST_DYLIB

      1. export TEST_DYLIB=`pwd`/Build/Release-iPhoneOS/testdylib/testmain

  4. Make sure ghidra_dir installed

    1. must be at least version 6

    2. in your ghidra install look for a folder that contains "ghidraRun"

    3. set $GH_PATH to the path specified above
    4. export GH_PATH=<path>
      1. use full not relative path
  5. Install the enterprise certificate
    1. In a finder window goto /Volumes/share/MDB/iOS/iOSFiles
    2. double click "Viper Ball Hunt Golf.developerprofile" 
      1. this will open xcode to import the profile
      2. the password is the file name minus .developerprofile
    3. double click ios_distribution.cer
    4. double click tag.p12
      1. might error importing but ignore the error
    5. double click testing.mobileprovision

List of tools now needed

  1. Xcode Developer Disk Image (DMG) 
    1. Install Xcode (get the lastest version released that should be on the MDB share /MDB/OSX/Software/Apple/Xcode/<newest version>)If it is not get with Inventory Manager
      1. find DMG in xcode directory 

        (/Applications/Xcode.app/Contents/Developer/Platforms/iPhoneOS.platform/DeviceSupport/<current version>/DeveloperDiskImage.dmg))

    2. put with the other DMG's ( redux/res/dmg )

Saline

  1. cd $IOS_REPO/saline
  2. make clean
  3. make
  4. run saline
    1.  $IOS_REPO//Dist/Release-MacOSX/saline $TEST_DYLIB

Adderall

  1. cd into $IOS_REPO/nightvision/python/
  2. open nv_kern_read_command.py in an editor 
    1. update kernel_info to include latest version number if not already present
    2. in get_ios_type, verify correct version number is in there
  3. update make files in $IOS_REPO/early-make by adding new prefix to new to the lists ( cd to $IOS_REPO/early-make and run subl . to open all for editing in sublime)
    1. early-bokken 
    2. early-close
    3. early-persist 
    4. early-remote 
  4. update rana
    1. update match function in $IOS_REPO/rana/ranas/<proper>_rana.py
    2. <proper> is most likely saline
    3. update means to ensure a verification is there for the current build version if not add another or  version.re is not none to the if statement
  5. cd $IOS_REPO/adderall
  6. make clean
  7. make
  8. under $OUT make a folder for adderall files
    1. mkdir $OUT/adder_files
    2. set $AD_OUT
      1. export AD_OUT=$OUT/adder_files
  9. cd $IOS_REPO/Dist/Release-MacOSX/
  10. ./adderall-dist $AD_OUT
  11. Hopefully have execution
  12. If theses are beta version then copy the folder in your $AD_OUT to: /Volumes/share/MDB/iOS/iOSFiles-Beta/
  13. If theses are not beta version then copy the folder in your $AD_OUT to: /Volumes/share/MDB/iOS/iOSFiles/

 

Symdra / Elsym

  1. under $OUT make a folder for adderall files
    1. mkdir $OUT/sym_files
    2. we will call it $SYM_OUT
    3. export SYM_OUT="$OUT/sym_files" 
  2. create branch for elsym-dist

    1. cd $IOS_REPO/elsym-dist/

    2. git checkout -b <descriptive_name>

  3. cd back to $IOS_REPO

    1. cd $IOS_REPO

  4. run symdra from the $IOS_REPO directory
    1. python symdra/ full -k $AD_OUT -a $SYM_OUT -g $GH_PATH -b <build_number_for_device>
      1. ex. "-b 13A4325c"
    2. (YES you are running python on a folder)
  5. When done symbol.db in $SYM_OUT
    1. ls $SYM_OUT/*/symbol.db
  6. This will be feed into Elsym
  7. Check that the symbol.db is good
    1. sqlite3 $SYM_OUT/<IPSW>/symbol.db
    2. select * from [ Master_Symbols ];
    3. .exit
  8. verify elsym
    1. cd $IOS_REPO/elsym
    2. git checkout -b <descriptive_name>
    3. el.py
      1. make sure build number is in the list
      2. might need to change "EL_CONFIG_LWVM_PARITION_FLAG_OFFSET"
        1. For iOS 8.4 we had to change it to 0x28. To determine:
          1. Open ghidra
          2. File -> Open Project
          3. goto the symdra_results/<IPSW>/Ghidra/kernelcache.gpr
          4. Open the System.kext and search for the string "disk0s1", "disk0s1s1", "Writable", and/or "UUID"
          5. This function is bsd_init (note this is heavily inlined and may not fully decompile)
          6. Find the call to _VNOP_IOCTL
          7. The second parameter is the ioctl number we need (in the past was 0xc01064c2 but might be broken into two instructions)
          8. Note the ioctl number
          9. Open com.apple.driver.LightweightVolumeManager.kext
          10. Search Program Text for the ioctl number (note you may want to search for half of it at a time) being used in a comparison
          11. This function is the ioctl handler
          12. Inside the true block of the conditional, look for the constants on the ioctl
          13. One of the commands sets a read-only field (1), the offset for this field is the LWVM_PARTION_FLAG_OFFSET.
    4. run and see what happen
      1. ZOO_IPSW_PATH=$SYM_OUT make copyonedist ZOO_TARGET=<IPSW>
        1. ex. ZOO_IPSW_PATH=$SYM_OUT make copyonedist ZOO_TARGET=iPad4,1_9.0_13A341
        2. _kernel_memory_allocate is only used for the development
    5. python verify.py $IOS_REPO/elsym-dist/
    6. python print_config.py $IOS_REPO/elsym-dist/el_<IPSW>.plist
  9. (folder: elsym) If changes to elsym were made put up for pull request
  10. (folder: elsym-dist) For new el_*.plist put up for up requests

Nightvision

  1. under $OUT make a folder for nightvision files
    1. mkdir $OUT/nv_files
    2. we will call it $NV_OUT
    3. export NV_OUT="$OUT/nv_files" 

  2. cd $IOS_REPO/nightvision
